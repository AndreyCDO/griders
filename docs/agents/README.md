# Griders Agent Team

Документация описывает виртуальную команду AI-агентов для работы с Griders. Основная точка входа для Codex - корневой `AGENTS.md`; этот раздел объясняет, как пользоваться ролями на практике.

## Какие агенты существуют

| Агент | За что отвечает | Когда использовать |
| --- | --- | --- |
| Product Manager / Orchestrator | Цель, маршрутизация, приоритеты, итог | Почти любая крупная задача или неопределенный запрос |
| Architect | Архитектура, границы модулей, риски изменений | Новые подсистемы, миграции, крупные изменения |
| Backend Developer | API, FastAPI, MCP, БД, webhook, интеграции | Серверная логика, Cryptorg/Bybit, MySQL, auth |
| Frontend Developer | Jinja/CSS/UI/UX | Страницы, личный кабинет, тарифы, мониторинг |
| Trading Specialist | Pine Script, GRID/DCA, TP/SL, сигналы | Изменения торговой механики |
| Strategy Skeptic | Независимая критика стратегий | Любые заявления о прибыльности или улучшении стратегии |
| Data Analyst | Метрики, backtest analysis, пользователи, тарифы | Данные, KPI, конверсии, статистика |
| Marketing Specialist | Позиционирование и тексты | Главная, тарифы, Telegram, onboarding |
| SEO Specialist | Органический поиск и public pages | SEO-план, metadata, sitemap, content plan |
| Technical Support Specialist | Пользовательские проблемы и ответы | Ошибки пользователей, объяснения, FAQ inputs |
| Reviewer | Независимая проверка изменений | После реализации кода или рискованных правок |
| DevOps Specialist | Deploy, env, Docker, server, rollback | Запуск, инфраструктура, production safety |
| Documentation Specialist | README, FAQ, инструкции | После изменений или повторяющихся вопросов |

## Независимые роли

- `Trading Specialist` не подтверждает прибыльность собственной стратегии.
- `Strategy Skeptic` должен быть независимым и искать причины, почему результат может не работать.
- `Reviewer` не должен быть единственным автором проверяемого кода.
- `Marketing Specialist` не утверждает цифры без `Data Analyst`.
- `Product Manager / Orchestrator` выбирает минимально необходимую команду, а не запускает всех подряд.

## Как запускается совместная работа

1. Orchestrator классифицирует задачу и называет нужные роли.
2. Сначала собираются факты из репозитория: файлы, контракты, данные, тесты.
3. Роли работают в стандартном маршруте из `docs/agents/workflows.md`.
4. Orchestrator объединяет выводы, отмечает противоречия и риски.
5. После реализации запускаются доступные проверки.
6. Итог владельцу проекта дается на русском языке: что сделано, что проверено, что осталось неизвестным.

## Как добавить нового агента

1. Создать `agents/<role-name>.md`.
2. Описать зону ответственности, правила, ограничения и формат вывода.
3. Добавить ссылку в корневой `AGENTS.md`.
4. Добавить строку в таблицу этого файла.
5. При необходимости добавить маршрут в `docs/agents/workflows.md`.
6. Проверить, что новая роль не дублирует существующую полностью.

## Как обновлять инструкции

- Обновляй инструкции вместе с изменением фактической архитектуры проекта.
- Не описывай будущие возможности как уже существующие.
- Если меняется торговая логика, обновляй ограничения для Trading Specialist, Strategy Skeptic и Data Analyst.
- Если меняются публичные страницы или тарифы, обновляй Marketing/SEO/Support/Documentation инструкции.
- Если меняется deploy model, обновляй DevOps и `DEPLOY.md`.

## Ограничения системы

- Агентская система не заменяет реальные данные Cryptorg, Bybit, TradingView и production logs.
- Прибыльность стратегии нельзя подтвердить только текстовым анализом.
- SEO-спрос и позиции требуют внешних SEO-инструментов.
- Конверсия, retention и тарифная эффективность требуют фактической аналитики.
- Production-действия, commit/push/merge и deploy требуют явного разрешения владельца.
